# Local Agent Implementation Checklist

> **Instructions for the Coding Assistant:** Before finalizing your implementation plan or returning control to the user, you MUST cross-reference your proposed logic and generated code against this checklist. Failure to verify these items will result in fatal structural errors.

### 1. Scaffold Integrity & Reuse
- [ ] **No Rewrites:** Did I firmly reject any impulses to rewrite the architecture from scratch? 
- [ ] **Light Modifications Only:** Did I copy the basic scaffold, modify as little as possible (since it works like a charm), and focus MOST edits strictly on `src/app.py`, `src/prompts.py`, `pyproject.toml`, `README.md`, and config YAML (`src/config_template.yaml`) to configure the required agent and subagent structures? Did I avoid editing engine files (`src/engine/`) unless there was genuinely no config toggle, format variable, or SDK-level alternative?
- [ ] **No Tool Rewrites:** Did I avoid rewriting existing tools (like files, web search, or parsing tools)? Did I prioritize reusing them as-is, make only minimal changes based on comments and documentation guidance, and reject writing new tool logic from scratch?
- [ ] **Prompt Preservation & Formatting Safety:** Did I retain all essential format keys (e.g., `{task_name}`, `{date}`, `{workspace_dir}`, `{delegation_instructions}`, `{delegate_tasks_quota}`, `{fetch_url_to_workspace_quota}`) in my customized prompts? Did I ensure sub-agent instructions contain `{task_name}` so child agents actually receive the task query, rather than running blindly?
- [ ] **Task Context Propagation:** Did I instruct the Orchestrator to analyze the user's query, break it down into specific tasks/angles, and pass *those specific angles* to sub-agents via the `delegate_tasks` tool (rather than passing generic or hardcoded instructions)?
- [ ] **String Interpolation Safety:** Did I avoid using single braces `{}` for un-interpolated placeholder variables inside system prompts? The engine uses a safe formatter that won't crash on unknown keys, but they will appear as literal `{text}` in the LLM prompt. Use double-braces `{{run_folder}}` or angle brackets `<run_folder>` for non-runtime placeholders.
- [ ] **No Pre-Formatting in app.py:** Did I avoid calling `.format()` on system prompt strings in `src/app.py` to prevent startup `KeyError` crashes, leaving formatting to the orchestration engine at runtime?
- [ ] **Anti-Looping Traps:** Did I inject a strict `<Anti-Looping>` clause into my `ORCHESTRATOR_INSTRUCTIONS` demanding that it NEVER call the exact same tool with identical arguments twice in a row? (Crucial to break `write_todos` infinite loops on small local models!).
- [ ] **Application Branding & Packaging:** Did I rename the default `basic-tui-agent` into a sensible, task-specific identity? This requires updating the `name` field in `pyproject.toml`, the `[project.scripts]` executable definition, AND the `APP_NAME`, `APP_TITLE`, and `APP_DESCRIPTION` constants at the top of `src/config.py`.
- [ ] **Import Preservation & Compatibility:** Did I ensure that I did not delete or rename variables/constants imported by the core engine (such as `ORCHESTRATOR_INSTRUCTIONS`, `SUBAGENT_INSTRUCTIONS`, or `SUBAGENT_DELEGATION_INSTRUCTIONS` from `src/prompts.py`)? If I renamed any prompts, did I preserve the original names as aliases pointing to the new variables to prevent fatal `ImportError` crashes at startup?
- [ ] **Quota Format Variable Naming:** Did I use the auto-populated `{tool_name_quota}` variables (e.g. `{web_search_quota}`, `{delegate_tasks_quota}`) in my prompts instead of inventing custom format keys that would require engine modifications? (See `PROMPTING.md` §2 for the naming convention).
- [ ] **No Duplicate YAML Keys:** Did I verify that every key under `settings.quotas` in `config_template.yaml` appears exactly once? (YAML silently drops duplicates — only the last value survives! Do NOT create separate "per-agent" quota sections with duplicate key names).
- [ ] **Shell Tool Pruning:** If the user did NOT explicitly ask for shell/bash execution, did I remove `run_shell_command` from the `WORKSPACE_TOOLS` array in `tools/__init__.py` and delete the import?

### 2. Sub-Agent Delegation (Strict Validation)
- [ ] **Location Restraint:** Are ALL of my sub-agents configured purely as `SubAgentConfig` definitions inside `src/app.py`? (Do not try to edit `engine/orchestrator.py` manually).
- [ ] **Nested Delegation Registry:** Did I register all nested sub-agents in the flat `sub_agents=[...]` list of the `AgentBuilder` in `src/app.py` instead of rewriting the orchestrator to pass configurations manually?
- [ ] **Dependency Ordering (CRITICAL):** Did I define deeply-nested sub-agents *before* their parent agents sequentially in the file? (e.g., defining `analyzer` above `searcher` so it can be passed into `tools=[]` without a `NameError`).
- [ ] **Sub-Agent Name Alignment:** Did I ensure the `name` defined in `SubAgentConfig` (e.g., `Analyzer`) exactly matches the target `agent_id` expected by the prompts and passed to the `delegate_tasks` tool (e.g., `agent_id="Analyzer"`)? (Do not invent name mismatches like registering the name as `"Analyzer"` but instructing the agent to delegate to `"page_analyzer"`).
- [ ] **One Prompt Per Sub-Agent Type (CRITICAL):** Does each sub-agent type have its OWN dedicated prompt constant in `prompts.py`? (e.g., `SEARCH_SUBAGENT_INSTRUCTIONS` for the Searcher, `ANALYZER_SUBAGENT_INSTRUCTIONS` for the Analyzer). Do NOT reuse a single `SUBAGENT_INSTRUCTIONS` for multiple agent types. Do NOT try to be clever by conditionally formatting a shared prompt. Each agent type gets a separate, self-contained prompt constant.
- [ ] **Selective Tool Lists Per Sub-Agent (CRITICAL):** Does each `SubAgentConfig` have its own **selective** tool list containing ONLY the tools that agent needs? Do NOT assign `WORKSPACE_TOOLS` (which includes everything) to specialized sub-agents. The Orchestrator, Searcher, and Analyzer must each have distinct tool sets. Tool separation is what FORCES delegation — if a parent has the child's tools, it will never delegate.
- [ ] **No Hallucinated State:** Am I statically hardcoding the `name="SubAgent"` in `client.as_agent()`? I MUST NOT invent global list counters (`_get_next_id()`).
- [ ] **Decorator Safety:** Does every delegation `@tool` have an explicit, strongly-typed `description` to prevent JSON schema generation failures?
- [ ] **Stream Execution:** Did I correctly wrap `sub_agent.run(..., stream=True)` and await `subagent_callback` to ensure the TUI doesn't freeze?

### 3. Tool Quotas & Execution
- [ ] **Clean Quota Configs (CRITICAL LIMIT CHECK):** Did I safely extract the integer limit from `config_template.yaml` instead of passing a raw `dict` dictionary? (e.g., extracting `.get("limit", 5)`). If you pass a dict into `tool_quotas_ctx.set(...)`, it will trigger a `TypeError` (int >= dict) causing `Error: Function failed.`!
- [ ] **Schema Omission Trap:** Did I actually inject the specific delegation tool into the sub-agent's `tools=[...]` array? (If the agent thinks about delegating but never calls the tool, it's because I forgot to attach the tool to it).
- [ ] **Architectural Boundaries:** Did I withhold inappropriate tools from the Orchestrator (e.g. stripping `web_search`) to force proper sub-agent delegation?
- [ ] **No Parser Tools:** Did I avoid exposing the utility functions inside `utils/parsers.py` (like `convert_to_markdown` or `extract_advanced_pdf`) directly to the agent as tools in `app.py`? (Ensure they are only used programmatically and internally under the hood).
- [ ] **Individual Tool Imports:** If I assigned different tool sets to different agents (e.g. withholding `web_search` from an Analyzer), did I import each individual tool from `tools` in `src/app.py`? (The scaffold example shows the import pattern: `from tools import read_workspace_file, web_search, ...`). Using a tool name in a `tools=[...]` list without importing it causes a fatal `NameError` at startup.

If the user asked for isolated workflow runs or per-session directories, did I set `settings.workspace.session_isolation: true` in `config_template.yaml` so that all files are transparently routed to timestamped run folders (e.g. `run_1748192400/`)?

### 5. Multi-Tier Data Flow Validation (For Nested Sub-Agent Architectures)
- [ ] **Agent-ID Routing in Prompts (CRITICAL):** Did I ensure that every agent's system prompt explicitly teaches the LLM which `agent_id` to pass when calling `delegate_tasks`? The LLM does not inherently know sub-agent names — the prompt must spell them out. For example, the Orchestrator's prompt must say: *"When delegating, you MUST pass `\"agent_id\": \"Searcher\"`"* and the Searcher's prompt must say: *"you MUST pass `\"agent_id\": \"Analyzer\"`"*. Without this, delegation silently routes to the wrong agent (defaults to `sub_agents[0]`).
- [ ] **Concrete Delegation Examples in Prompts:** Did I include a concrete `delegate_tasks` call example (showing the JSON shape with `task_name`, `instructions`, and `agent_id`) directly in the system prompt of every agent that delegates? This prevents the LLM from guessing the schema and omitting `agent_id`.
- [ ] **Data-Flow Integrity (Fetch → Capture → Forward):** For each parent→child delegation in a nested pipeline, did I verify that the parent's prompt explicitly instructs the LLM to: (1) capture relevant output from its own tools (e.g., the filename returned by `fetch_url_to_workspace`), and (2) embed that output into the delegation `instructions` string for the child agent? If the child agent needs to read a file, the parent MUST pass the exact filename — children cannot read files that were never fetched or whose names they don't know.
- [ ] **Tool Consistency with Delegation Model:** If an agent is supposed to delegate a capability (e.g., file reading) to a child, did I verify that the parent does NOT possess the tools that the child is supposed to use? Giving a parent both `fetch_url_to_workspace` AND `read_workspace_file` defeats delegation — the LLM will read files itself instead of delegating, bloating its own context window and bypassing the child agent entirely.
- [ ] **Double-Brace JSON Examples (CRITICAL):** If I embedded concrete JSON delegation examples (e.g., `delegate_tasks(tasks=[...])`) inside Python prompt strings in `src/prompts.py`, did I use double-braces `{{` and `}}` for all JSON curly braces? Single braces `{` are reserved for engine format variables like `{date}` and `{task_name}` — using them for JSON causes a fatal `ValueError: Invalid format specifier` crash at runtime. (See `PROMPTING.md` §6 and §8).
- [ ] **Proportional Effort & Stop Criteria:** Did I include prompt instructions that match the agent's effort to task complexity? Simple tasks should not trigger heavy multi-step plans or multiple delegations. Did I also teach the agent when it has enough information to stop (e.g., stop after finding the answer from a reliable source, don't exhaust all tool quotas)? Without these, agents over-plan trivial requests and under-deliver on complex ones.

### 6. Evaluation (When Requested)
- [ ] **Eval Harness Setup:** If the user asks for evaluation or benchmarking, did I copy `eval/evaluate.py` and `eval/eval_config.yaml` from the scaffold template (`examples/basic-tui-agent/eval/`) into the project's `eval/` directory?
- [ ] **Agent-Specific Dataset:** Did I create a `eval/dataset.jsonl` tailored to this specific agent — not a generic placeholder? Each item must have a `query`, `criteria`, and an appropriate `eval_type` (`contains`, `regex`, or `llm_judge`). See `EVALUATION.md` for the format.
- [ ] **Artifact vs Stdout:** For agents that write structured output files, did I set `"artifact": "filename.md"` in the dataset so the harness reads the file rather than stdout? Simple factual queries can use `"artifact": null` (stdout).
- [ ] **Headless Mode Verified:** Did I confirm the agent runs cleanly with `python src/app.py --prompt "..." --auto-approve` before setting up the eval harness? The harness depends entirely on this working without interactive prompts.
- [ ] **Workspace Patched for Disk:** Did I note (in the README or eval/README.md) that the harness patches the agent config to disk-backed workspace for each run? Users do not need to change `config_template.yaml` manually — the harness overrides it.
