# Example Prompts

The following are example prompts you can use to instruct a coding assistant to build powerful, complex agents using the `local-agent-builder` skill.

---

## 1. Deep Research Agent (Multi-Tier Hierarchical Delegation)

**Copy and paste the following prompt to your coding assistant:**

```text
Using the local-agent-builder skill, build a Deep Research Agent that can perform extensive web-based research and document analysis. 

Agent Architecture

1. Orchestrator Agent:
   - Scope: Serves as the primary task manager and reporter. It breaks down the main user query into actionable research tasks and synthesizes the final report.
   - Instructions: Creates a TODO list outlining the research phases. For each phase, it dispatches specialized Search Sub-Agents to gather information. If multiple independent research angles exist for a phase, it MUST delegate those Search Sub-Agents concurrently rather than sequentially. It must write the user's original query to query.md and produce a final final_report.md with a highly professional structure.
   - Tooling Guidelines: Provide tools for task management and file writing (e.g., `write_todos`, `read_todos`, `write_workspace_file`, `read_workspace_file`, `list_workspace_files`). Do not give this agent web search or URL-fetching capabilities, as that breaks the delegation model and forces the orchestrator to do the heavy lifting.
   - Delegation Routing: The Orchestrator's system prompt MUST explicitly teach the LLM to pass `"agent_id": "Searcher"` in every `delegate_tasks` call. Without this, delegation routes to the wrong sub-agent. Include a concrete example in the prompt, like:
     ```
     When delegating research tasks, you MUST always specify the agent:
     delegate_tasks(tasks=[
       {{"task_name": "Research topic X", "instructions": "Search for ...", "agent_id": "Searcher"}},
       {{"task_name": "Research topic Y", "instructions": "Search for ...", "agent_id": "Searcher"}}
     ])
     ```
     IMPORTANT: Note the double-braces `{{` and `}}` above. Since these examples are embedded inside Python prompt strings that go through `.format()`, you MUST double-brace all JSON curly braces to prevent `ValueError: Invalid format specifier` crashes. Single braces `{` are reserved for engine variables like `{date}` and `{task_name}`.

2. Search Sub-Agent:
   - Scope: Acts as the web crawler to find relevant pages and download them for deeper analysis.
   - Instructions: Receives the specific research task from the Orchestrator. It MUST NOT hallucinate or guess URLs from its internal knowledge. The strict workflow must be:
     1. Use `web_search` to find live, verified links.
     2. Fetch those specific links to the local workspace via `fetch_url_to_workspace`.
     3. CRITICAL DATA-FLOW RULE: After fetching, the tool returns a confirmation message containing the saved filename (e.g., "Fetched URL successfully to 'microsoft_ai_research_143022.md'"). The Search Sub-Agent MUST capture this exact filename and pass it in the delegation instructions to the Page Analyzer. If it does not pass the filename, the Analyzer will not know which file to read.
     4. Delegate the reading and parsing of all downloaded documents concurrently to a batch of nested Page Analyzer Sub-Agents, passing each the exact filename and the original research context.
     5. When returning findings, explicitly include all URLs and sources so the Orchestrator can cite them.
   - Delegation Routing: The Search Sub-Agent's system prompt MUST explicitly teach the LLM to pass `"agent_id": "Analyzer"` in every `delegate_tasks` call. Include a concrete example in the prompt, like:
     ```
     After fetching pages, delegate analysis. You MUST pass the exact filename and agent:
     delegate_tasks(tasks=[
       {{"task_name": "Analyze microsoft_ai_research_143022.md", "instructions": "Read the file 'microsoft_ai_research_143022.md' and extract key findings about ...", "agent_id": "Analyzer"}}
     ])
     ```
     IMPORTANT: Use double-braces `{{` and `}}` for all JSON curly braces in prompt strings. Single braces are reserved for engine format variables.
   - Tooling Guidelines: ONLY give this agent web capabilities and a tool to dispatch analyzers (e.g., `web_search`, `fetch_url_to_workspace`). It MUST NOT have `read_workspace_file` or `grep_workspace_file` — withholding these tools forces delegation to the Analyzer. If the Searcher can read files itself, it will skip delegation and bloat its own context window, defeating the entire architecture.

3. Page Analyzer Sub-Agent:
   - Scope: Focuses exclusively on reading and extracting precise data from individual documents already downloaded to the local workspace.
   - Instructions: Receives the original research context and the EXACT FILENAME of a specific document from the Search Sub-Agent. If the plaintext page is large, it MUST use `grep_workspace_file` to locate specific target keywords (like dates, names, or "born"). Once the relevant line numbers are identified, it can use `read_workspace_file` with precise `start_line` and `end_line` bounds to read the surrounding context efficiently. It MUST extract and surface any relevant "Read More" links, navigation menus, or references explicitly back to the Search Sub-Agent, allowing the parent Search Sub-Agent to continue diving down rabbit holes if necessary.
   - Tooling Guidelines: Only equip this agent with local file parsing tools (e.g., `read_workspace_file`, `grep_workspace_file`). It should have zero web-fetching capabilities, ensuring it stays focused purely on reading the data already on the disk.

Operational Requirements
- Strict Directory Path Management: The application code MUST handle directory isolation automatically:
  1. Create a timestamped isolated folder per run/session (e.g., `run_TIMESTAMP/`).
  2. File tools MUST automatically map all read and write operations into this run folder.
  3. Agents MUST NOT be aware of the run folder. They should read and write files directly (e.g., `query.md`, `final_report.md`).
  4. Downloaded pages MUST be named using a url-slug + timestamp to prevent collisions (e.g., `microsoft_ai_research_143022.md`). NEVER allow the creation of separate subfolders.
- Anti-Looping Traps: You MUST explicitly bake the `<Anti-Looping>` XML block (from PROMPTING.md) into the system instructions of BOTH the Orchestrator and every Sub-Agent to prevent infinite retry loops on tool failures.
- Stop Early Conditions: You MUST explicitly instruct the Orchestrator, Search Sub-Agent, and Page Analyzer to STOP working immediately when they believe they have enough information to confidently answer the user's query. Explicitly tell them: "Do NOT visit all links or max out your quotas. If you have found the answer corroborated by at least 2 strong sources, stop immediately and return your findings."
- Adaptive Reporting: Instruct the Orchestrator to dynamically determine the `final_report.md` template structure (simple list vs. complex sections like Methodology/Sources) based strictly on the complexity of the original user query. 
- Configurable Quotas: Every single tool across all agents (including sub-agent delegations) MUST be heavily restricted with quotas in the config.yaml to prevent infinite web-scraping recursion.

CRITICAL ANTI-PATTERNS TO AVOID:
1. DO NOT give a parent agent the tools that its child agent is supposed to use. If the Searcher can `read_workspace_file`, it will never delegate to the Analyzer.
2. DO NOT omit `agent_id` from delegation calls. Without explicit routing, delegation defaults to the first registered sub-agent (index 0), which is likely the wrong agent.
3. DO NOT assume files exist before they are fetched. The Analyzer can only read files that the Searcher has already downloaded. The Searcher's prompt MUST instruct it to capture the filename from the fetch tool's response and pass it to the Analyzer.
```
