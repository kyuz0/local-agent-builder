# Example Prompts

The following are example prompts you can use to instruct a coding assistant to build powerful, complex agents using the `local-agent-builder` skill.

---

## 1. Deep Research Agent (Multi-Tier Hierarchical Delegation)

**Copy and paste the following prompt to your coding assistant:**

```text
Using the local-agent-builder skill, build a Deep Research Agent that can perform extensive web-based research and document analysis. 

Agent Architecture — Delegation Chain & Tool Assignment

DELEGATION CHAIN (strictly hierarchical, one direction only):
  Orchestrator  →  delegates to  →  Searcher  →  delegates to  →  Analyzer

TOOL ASSIGNMENT TABLE (each agent gets ONLY these tools, no exceptions):
  Orchestrator : write_workspace_file, list_workspace_files, write_todos, read_todos, think_tool, delegate_tasks (auto-injected)
  Searcher     : web_search, fetch_url_to_workspace, think_tool, delegate_tasks (auto-injected)
  Analyzer     : read_workspace_file, grep_workspace_file, think_tool

The Orchestrator has NO web tools and NO file reading tools. It delegates to Searchers.
The Searcher has NO file reading tools. It delegates to Analyzers.
The Analyzer has NO web tools and NO delegation capability. It reads files and returns findings.

This separation is enforced TWO ways in app.py:
1. Tool withholding: each SubAgentConfig gets only its own selective tool list.
2. Scoped sub_agents: each SubAgentConfig declares which children it can delegate to via sub_agents=[...].
   The engine only injects delegate_tasks into agents that have children. Leaf agents cannot delegate.
   The Orchestrator's sub_agents=[searcher]. The Searcher's sub_agents=[analyzer]. The Analyzer has no sub_agents.

1. Orchestrator Agent:
   - Scope: Primary task manager and final report writer. It plans research, dispatches Searchers, and synthesizes their returned summaries into `final_report.md`.
   - Tools: `write_workspace_file`, `list_workspace_files`, `write_todos`, `read_todos`, `think_tool`, `delegate_tasks`. It has NO `read_workspace_file`, NO `web_search`, NO `fetch_url_to_workspace`.
   - Instructions: Creates a TODO list, dispatches Searcher sub-agents (concurrently when independent), and writes the final report. It MUST wait for Searcher results — it cannot read downloaded files itself.
   - Delegation: The Orchestrator delegates ONLY to the Searcher (agent_id: "Searcher"). It does NOT know about the Analyzer and MUST NOT delegate to it. The Orchestrator's system prompt MUST include a concrete `delegate_tasks` call example with `agent_id: "Searcher"`. (See PROMPTING.md §8.)

2. Search Sub-Agent (Searcher):
   - Scope: Web crawler that finds and downloads pages, then delegates their analysis to Analyzers.
   - Tools: `web_search`, `fetch_url_to_workspace`, `think_tool`, `delegate_tasks`. It has NO `read_workspace_file`, NO `grep_workspace_file`.
   - Instructions: Receives a research task from the Orchestrator. It searches the web, fetches URLs to the workspace, captures the returned filenames, then delegates each file to an Analyzer for reading.
   - Delegation: The Searcher delegates ONLY to the Analyzer (agent_id: "Analyzer"). Its system prompt MUST include a concrete `delegate_tasks` call example with `agent_id: "Analyzer"`. (See PROMPTING.md §8.)
   - Data-Flow Integrity: After `fetch_url_to_workspace` returns a filename, the Searcher MUST embed that exact filename in the delegation instructions to the Analyzer. Without this, the Analyzer won't know which file to read. (See PROMPTING.md §9.)
   - Because the Searcher has NO file reading tools, it is physically forced to delegate to the Analyzer. This is intentional — it prevents the Searcher from reading large files and blowing its own context.

3. Page Analyzer Sub-Agent (Analyzer):
   - Scope: Reads and extracts data from individual documents already downloaded to the workspace.
   - Tools: `read_workspace_file`, `grep_workspace_file`, `think_tool`. It has NO `web_search`, NO `fetch_url_to_workspace`, NO `delegate_tasks`.
   - Instructions: Receives the exact filename and research context from the Searcher. Uses `grep_workspace_file` to locate keywords, then `read_workspace_file` with precise line ranges. Returns a concise summary of findings and any relevant links back to the Searcher.
   - The Analyzer is a leaf node — it cannot delegate further and cannot fetch new URLs.

Operational Requirements
- Strict Directory Path Management: The application code MUST handle directory isolation automatically:
  1. Create a timestamped isolated folder per run/session (e.g., `run_TIMESTAMP/`).
  2. File tools MUST automatically map all read and write operations into this run folder.
  3. Agents MUST NOT be aware of the run folder. They should read and write files directly (e.g., `query.md`, `final_report.md`).
  4. Downloaded pages MUST be named using a url-slug + timestamp to prevent collisions (e.g., `microsoft_ai_research_143022.md`). NEVER allow the creation of separate subfolders.
- Anti-Looping Traps: You MUST explicitly bake the `<Anti-Looping>` XML block (from PROMPTING.md) into the system instructions of BOTH the Orchestrator and every Sub-Agent to prevent infinite retry loops on tool failures.
- Stop Early Conditions: You MUST explicitly instruct the Orchestrator, Search Sub-Agent, and Page Analyzer to STOP working immediately when they believe they have enough information to confidently answer the user's query. Explicitly tell them: "Do NOT visit all links or max out your quotas. If you have found the answer corroborated by at least 2 strong sources, stop immediately and return your findings."
- Proportional Search Depth: The Orchestrator MUST assess query complexity BEFORE planning its delegation strategy:
   - Simple factual queries (single fact lookups): dispatch a SINGLE Searcher. One or two authoritative sources is sufficient. Do NOT create multi-phase research plans for simple lookups.
   - Multi-fact queries (multiple facts likely on the same page): a single Searcher is still sufficient.
   - Comparative or synthesis queries: dispatch one Searcher per independent research angle, concurrently.
   - Deep research / report generation: use the full multi-phase approach.
- Source Quality Awareness: The Search Sub-Agent MUST evaluate source authority to decide when to stop:
   - Authoritative/official sources (manufacturer websites, official documentation, spec sheets): ONE source is sufficient. Do not search further to corroborate an official spec page.
   - Semi-authoritative sources (established tech publications): one source is usually sufficient, but a second is welcome if readily available.
   - Informal sources (forums, blogs, wikis): corroborate with at least one additional source before trusting the data.
   For simple factual queries, if the first web search returns a clear answer from an authoritative source, fetch that ONE page, delegate analysis, and stop. Do not run additional searches.
- Adaptive Reporting: Instruct the Orchestrator to dynamically determine the `final_report.md` template structure (simple list vs. complex sections like Methodology/Sources) based strictly on the complexity of the original user query.
- Configurable Quotas: Every single tool across all agents (including sub-agent delegations) MUST be heavily restricted with quotas in the config.yaml to prevent infinite web-scraping recursion.

CRITICAL ANTI-PATTERNS TO AVOID:
1. DO NOT give a parent agent the tools that its child agent is supposed to use. If the Searcher can `read_workspace_file`, it will never delegate to the Analyzer. If the Orchestrator has `web_search`, it will never delegate to the Searcher.
2. DO NOT let the Orchestrator delegate directly to the Analyzer. The Orchestrator only knows about the Searcher. The Searcher only knows about the Analyzer. This is a strict chain.
3. DO NOT omit `agent_id` from delegation calls. Without explicit routing, delegation defaults to the first registered sub-agent (index 0), which is likely the wrong agent.
4. DO NOT assume files exist before they are fetched. The Analyzer can only read files that the Searcher has already downloaded. The Searcher's prompt MUST instruct it to capture the filename from the fetch tool's response and pass it to the Analyzer.
```

## 2. Agentic RAG Agent (Document Corpus Search & Analysis)

**Copy and paste the following prompt to your coding assistant:**

```text
Using the local-agent-builder skill, build an Agentic RAG Agent that can answer
questions about a local document library using semantic search and precise
document reading.

Agent Architecture

1. Single Flat Agent (no sub-agents needed for this pattern):
   - Scope: Answers user questions by searching a pre-indexed document corpus
     using semantic similarity, keyword matching, and precise document reading.
   - Workflow:
     1. Use `semantic_search` to find relevant chunks via vector similarity.
     2. If the query contains specific terms (IPs, names, exact IDs), also use
        `keyword_search` for exact matches.
     3. Use `list_library_files` to discover what documents are available.
     4. Use `read_library_file` with precise line bounds to read context around
        found chunks.
     5. Synthesize findings and cite sources with inline [1], [2] markers.
   - Tooling: Give the agent `semantic_search`, `keyword_search`,
     `list_library_files`, `read_library_file`, and `think_tool`.
     Do NOT give it web search or URL-fetching tools — this is a pure
     read-only corpus query agent.

CRITICAL: Document Ingestion is NOT an LLM Activity
   - The vector database is populated at STARTUP, before the LLM loop begins.
   - The application code in `app.py` must:
     1. Read RAG config from `settings.rag` (library_dir, chunk_size, etc.)
     2. Instantiate VectorStore from `src/utils/vectorstore.py`
     3. Call `vectorstore.sync()` to scan for new/changed files, parse them
        (liteparse preferred, markitdown fallback), chunk via semchunk, embed
        via /v1/embeddings, and store in sqlite-vec
     4. Call `init_rag_tools(vectorstore)` to wire the instance into the tools
     5. Pass RAG tools to the AgentBuilder
   - The LLM agent NEVER triggers embedding or ingestion — it only queries
     the pre-built index via its tools.

RAG Library Configuration
   - The user points the agent to a document directory via
     `settings.rag.library_dir` in config.yaml.
     Default: `~/.{APP_NAME}/libraries/`
   - The sqlite-vec database is stored as `.rag.db` inside that directory.
   - Embedding endpoint defaults to the same host as the LLM, but can be
     overridden via `api.embedding_base_url` in config.yaml.
   - Chunk size and overlap are configurable via `settings.rag.chunk_size`
     and `settings.rag.chunk_overlap`.
   - If RAG is not needed, remove `sqlite-vec` and `semchunk` from deps,
     delete `src/tools/rag.py` and `src/utils/vectorstore.py`, and remove
     the embedding and RAG config blocks from config_template.yaml.

Operational Requirements
   - Anti-Looping: Bake anti-looping blocks into the agent's prompt.
   - Stop Early: If semantic_search returns a clear answer, cite it and stop.
     Do not read every document exhaustively.
   - Quotas: Restrict all RAG tools with quotas to prevent search loops.
   - Citation: Agent MUST cite sources using inline [1], [2] markers and
     list exact filenames at the end of its response.
```
